# ros2/

ROS 2 Jazzy workspace used on the **dev workstation** — bringup, MoveIt
configs, joint-state publishers, robot description, etc. This is *not* what
runs on the Jetson at runtime (see `jetson-agent/` for that).

The container also acts as the **micro-ROS agent host**: it bridges the
Teensy locomotion controller (firmware in `firmware/bebop-locomotion/`,
running the `[env:ros]` build) to the rest of the ROS 2 graph via USB
serial.

## Layout

| Path                          | What it is                                                       |
|-------------------------------|------------------------------------------------------------------|
| `src/bebop_pilot/`            | High-level pilot / behaviour package.                            |
| `src/bebopv2_description/`    | URDF + meshes + xacro for the robot.                             |
| `src/ros2.repos`              | `vcstool` manifest for additional source dependencies.           |
| `docker/Dockerfile`           | Jazzy-desktop-full base + nav2 / MoveIt / ros2_control / ros-gz. |
| `docker/entrypoint.sh`        | Bootstraps micro-ROS, builds the workspace, sources the overlay. |
| `docker/requirements.txt`     | Python deps installed into the container's `/opt/venv`.          |

`build/`, `install/`, `log/`, and `src/{micro_ros_setup,uros}/` are
gitignored — they're regenerated by the entrypoint or `colcon`.

## Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│              Dev workstation — bebop_ros2 container                │
│                                                                    │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────────────┐    │
│  │ bebop_pilot  │  │ teleop / ...  │  │ Nav2 / MoveIt / ...  │    │
│  └──────┬───────┘  └───────┬───────┘  └──────────┬───────────┘    │
│         │                  │                     │                │
│         └────────── DDS (RMW_FASTRTPS) ──────────┘                │
│                            │                                      │
│                ┌───────────┴────────────┐                         │
│                │   micro_ros_agent      │                         │
│                │   serial_dev:/dev/ttyACM0                        │
│                └───────────┬────────────┘                         │
└────────────────────────────┼────────────────────────────────────--┘
                             │ USB Serial (bind-mount /dev)
                             │
            ┌────────────────▼────────────────────┐
            │     Teensy 4.1 (firmware/bebop-     │
            │     locomotion, env=ros)            │
            │                                     │
            │     /dev/ttyACM0  micro-ROS client  │
            │     /dev/ttyACM1  debug serial      │
            └─────────────────────────────────────┘
```

Both Isaac containers and `bebop_ros2` share `ROS_DOMAIN_ID=0` and
`RMW_IMPLEMENTATION=rmw_fastrtps_cpp` so topics flow across containers
on the same host.

## Why the workspace is host-side

`src/` lives on the host and is bind-mounted into the container at
`/home/$USER/master_ros2_ws/src`. That way `colcon build` artifacts
(`build/`, `install/`, `log/`) can stay inside the container while you
edit packages from your host IDE.

The container is named `bebop_ros2` and joins both the `sim` and `lab`
docker-compose profiles, so any time you bring up Isaac Sim or Isaac Lab,
the ROS 2 dev container comes along with it.

## Common commands

From the repo root:

```sh
just sim-up         # bring up isaac_sim + bebop_ros2 (profile: sim)
just lab-up         # bring up isaac_lab + bebop_ros2 (profile: lab)
just sim-down       # tear down sim profile
just lab-down       # tear down lab profile
just ros2-build     # rebuild only the bebop_ros2 image
just ros2-shell     # interactive shell in the running bebop_ros2
```

Inside the container (after the entrypoint runs):

```sh
cd /home/$USER/master_ros2_ws
colcon build --symlink-install
source install/setup.bash
ros2 launch bebop_pilot ...
```

## Development workflow

End-to-end loop when working with real Teensy hardware:

```sh
# 1. Build + flash the locomotion firmware on the host (PlatformIO).
just fw-build           # firmware/bebop-locomotion env=ros
just fw-flash

# 2. Bring up the dev containers. First run takes ~2 minutes because the
#    entrypoint clones and builds the micro-ROS agent (see below).
just lab-up             # or just sim-up

# 3. Open an interactive shell in the running ros2 container.
just ros2-shell

# 4. Inside the container — the workspace is already sourced.
ros2 launch bebop_pilot bringup.launch.py
ros2 topic echo /joint_states

# 5. (Optional) Tail the Teensy debug serial from the host.
cat /dev/ttyACM1
```

`/dev` is bind-mounted into the container, so `/dev/ttyACM*` shows up
inside without any extra config — adding/removing the Teensy while the
container is running works.

## micro-ROS agent

The `bebop_ros2` container bootstraps and hosts the micro-ROS agent
automatically. `docker/entrypoint.sh` does this on first run:

1. Clones [`micro_ros_setup`](https://github.com/micro-ROS/micro-ROS/tree/main/utility/micro_ros_setup)
   (Jazzy branch) into `src/micro_ros_setup/`.
2. Runs `rosdep update && rosdep install --from-paths src --ignore-src -y`.
3. `colcon build --packages-select micro_ros_setup --symlink-install`.
4. `ros2 run micro_ros_setup create_agent_ws.sh` (creates `src/uros/`).
5. `ros2 run micro_ros_setup build_agent.sh` (~2 minutes; populates
   `install/micro_ros_agent/`).

Both `src/micro_ros_setup/` and `src/uros/` are gitignored — they're
regenerated whenever a fresh container starts and the host paths are
empty.

After bootstrap, run the agent against the Teensy:

```sh
# Inside the container:
ros2 run micro_ros_agent micro_ros_agent serial \
    --dev /dev/ttyACM0 --baudrate 115200
```

### Re-bootstrapping by hand

If the entrypoint's heuristic ever bails out (e.g. you partially deleted
something), do it manually:

```sh
cd /home/$USER/master_ros2_ws
rm -rf src/micro_ros_setup src/uros build install log

git clone -b jazzy \
    https://github.com/micro-ROS/micro_ros_setup.git src/micro_ros_setup
sudo rosdep update
sudo rosdep install --from-paths src --ignore-src -y
colcon build --packages-select micro_ros_setup --symlink-install
source install/setup.bash

ros2 run micro_ros_setup create_agent_ws.sh
ros2 run micro_ros_setup build_agent.sh
source install/setup.bash
```

## Troubleshooting

### Container can't see the Teensy

```sh
ls /dev/ttyACM*        # on the host AND inside the container
```

If it shows on the host but not in the container, the `/dev` bind-mount
likely failed — `docker compose down && just lab-up` again. The compose
file already grants the right `device_cgroup_rules` for `ttyACM*`
(major 166).

### Agent starts but the Teensy doesn't connect

- Confirm the Teensy is on `/dev/ttyACM0` and the firmware is the
  `[env:ros]` build (the `[env:sim]` build talks micro-ROS to Isaac, not
  to the agent).
- The Teensy retries the agent handshake for ~50 seconds at boot;
  status LED blinks while waiting.
- Tail debug output: `cat /dev/ttyACM1` (or `pio device monitor --port /dev/ttyACM1`
  on the host).

### `ros2 node list` doesn't show the firmware node

- Confirm the agent is actually running: `pgrep -a micro_ros_agent` inside
  the container.
- Confirm `ROS_DOMAIN_ID` matches across host, container, and Teensy
  build (the container exports `ROS_DOMAIN_ID=0`; the Teensy build's
  default needs to match).

### `colcon build` complains about already-built micro-ROS

The micro-ROS agent build artifacts live in `install/micro_ros_agent/`.
A clean rebuild is just `rm -rf build install log && colcon build`.
